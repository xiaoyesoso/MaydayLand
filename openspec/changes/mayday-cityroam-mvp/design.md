## Context

MaydayLand 既有仓库 `MaydaySkills/` 是面向 LLM Agent 的 Skill 资源包集合（13 个 Skill，覆盖歌词、和弦、知识库、电台等），不直接服务 C 端用户。本次变更引入面向 C 端的 **微信小程序** 产品形态：「五月天·城市漫游」，把 Skill 仓库沉淀的歌词数据（9 张专辑 / 120 首）转化为"歌词地图 + 打卡卡片 + 五迷暗号"的真实生活娱乐场景。

约束：
- 仓库已采用渐进式加载、版权安全（≤ 1–2 句歌词片段）等设计原则（参见 `MaydaySkills/AGENTS.md`、`docs/sdd/30-quality.md`），新产品 **MUST** 沿用这些底线。
- MVP 强调最低成本快速上线：不自建后端服务，只用微信原生 + CDN 静态 + 微信云开发（少量云函数）。
- 团队规模与时间表见 `五月天粉丝互动MVP需求文档.md` §6 上线与迭代。

利益相关者：五月天粉丝（C 端）、合作店铺（B 端）、运营团队（数据 / 审核）、版权代理方。

## 初赛评审维度对齐

本次变更以「交互式 HTML Demo」作为比赛评审交付物，所有设计决策需对齐初赛 4 个评审维度（创新性 30%、实用性 30%、完成度 20%、美观度/设计体验 20%）。

| 维度 | 权重 | 设计决策映射 |
|------|------|--------------|
| **创新性** | 30% | D1 采用 HTML Demo 零依赖单文件交付，D3 复用 MaydaySkills 歌词数据，D4 一次性 Token 暗号核销，D7 卡片图作为分享封面——均服务于"歌词 + LBS + 同好认证"的差异化创新 |
| **实用性** | 30% | D2 CDN 静态 + 云开发降低落地成本，D5 双圈到店核验兼顾城市 GPS 漂移，D6 用户照片完全本地不上云，确保 Demo 可真实运行且隐私合规 |
| **完成度** | 20% | D9 明确比赛交付形态为单文件 HTML Demo，双击即开；D8 UGC 审核状态机保证闭环；所有核心链路在浏览器内可完整操作 |
| **美观度/设计体验** | 20% | D9 视觉系统强制五球配色 + 卜卜圆润几何 + STAYREAL 潮玩贴纸感；启动页动画、页面转场、卡片模板切换均需体现 |

## Goals / Non-Goals

**Goals**
- 在不引入复杂后端服务的前提下，交付 5 个核心 capability：`lyric-corner-map / checkin-card / fan-passcode / concert-hub / share-growth`，并配套数据 pipeline `lyric-data-pipeline`。
- 5 个核心城市 × 50 个种子角落 + 5–10 家首批合作店铺，构建可验证 MVP。
- 验证假设：分享率 ≥ 30%、角落打卡率 ≥ 25%、UGC 共创 ≥ 500 个 / 月。
- 完整复用 MaydaySkills 既有歌词数据资产，不引入新的歌词数据来源。

**Non-Goals**
- **不**实现实时弹幕同步、IM 聊天、任意时长 AI 电台（属于 SK9 mayday-radio 的范畴）。
- **不**引入会员等级、积分、商城、支付能力。
- **不**自建后端服务（不上 Node/Spring/Go），仅使用 CDN + 微信云开发。
- **不**做完整 AI 歌词推荐（MVP 用预置标签匹配；AI 化在后续阶段评估）。
- **不**接入第三方数据埋点平台（神策 / 友盟），先用微信原生分析。

## Decisions

### D1：技术选型 — 微信小程序原生 vs Taro/uni-app
**选择**：原生（WXML/WXSS/JS + canvas 2d）作为正式上线形态；**本次比赛交付采用交互式 HTML Demo**（原生 HTML5 + CSS3 + Vanilla JS + Canvas 2D，单文件自包含）。
**理由**：MVP 阶段对包大小与性能敏感；`canvas 2d` 在原生 SDK 中支持最稳定；团队对 Taro 编译产物的体积无控制把握。比赛要求"交互式可体验 HTML 格式文件 Zip 打包上传"，故以 HTML Demo 作为可点击评审交付物，浏览器双击即开，零依赖；正式上线仍走微信原生。
**备选**：Taro 4.x — 长期跨端价值高，但 MVP 不需要支付/H5 跨端，引入收益不显著。

### D2：后端形态 — CDN 静态 + 微信云开发 vs 自建服务
**选择**：CDN 托管所有可公开数据（`corners.json`、`schedule.json`、`news.json`、`lyric-snippets.json`），微信云开发只承载 3 个云函数（`createPasscodeToken / verifyPasscodeToken / submitUserCorner`）+ 3 个集合（`user_corners / share_events / user_badges / checkin_archive`）。
**理由**：CDN + 静态 JSON 满足读多写少场景（90%+ 流量），冷启动 ≤ 200ms；云开发 serverless 模式天然适配 MVP 流量曲线，零运维。
**备选**：自建 Node + MongoDB — 成本高且 MVP 不需要复杂查询；Strapi/Directus — 对运营友好但部署 / 鉴权成本高。

### D3：歌词数据来源 — 复用 MaydaySkills 单一主源
**选择**：以 `MaydaySkills/mayday-mood/references/lyrics-db/` 为唯一主源，通过 `scripts/sync-lyrics-snippets.py` 抽片段，**MUST NOT** 引入第二份歌词数据来源。
**理由**：避免与既有 Skill 仓库出现"歌词双源不一致"风险；既有 Skill 已通过质量门禁，复用 = 复用合规校验。
**备选**：网易云 / QQ 音乐 API 抓取 — 版权风险高，被否决。

### D4：暗号核销机制 — 一次性 Token 二维码
**选择**：用户端云函数生成短 TTL（10 分钟）一次性 token，渲染二维码；商家端使用 `wx.scanCode` 扫码，云函数原子校验 + 记录。
**理由**：避免静态暗号被截图传播；不需要复杂 OAuth；商家端只需在小程序内开通商家分包。
**备选**：四位静态数字暗号（朋友间口口相传）— 易泄漏，无法做核销与权益限额。

### D5：到店核验 — 200m 严格 + 1km 宽松降级
**选择**：默认 200m 严格圈，失败时允许 1km 宽松圈但卡片标注"附近打卡"，超过 1km 拒绝。
**理由**：城市建筑 GPS 漂移常见，硬性 200m 会损害体验；宽松圈 1km 大于一般街区距离，避免远端薅羊毛。
**备选**：固定 500m — 折中但无法区分"现场"与"附近"两种语义。

### D6：用户照片处理 — 完全本地，零上云
**选择**：拍照后 `tempFilePath` 仅在 Canvas 合成时使用，最终输出的卡片图也仅在本地 / 通过微信原生分享 API 流转，**MUST NOT** 通过 `wx.uploadFile` 上传到任何对象存储。
**理由**：合规优先（避免肖像权 / 隐私争议），同时大幅降低存储成本。
**风险**：用户换设备无法找回历史卡片 — 通过"打卡足迹"只存元数据 + 缩略图（≤ 50KB）+ cornerId 解决。

### D7：分享路径 — 卡片图作为 imageUrl
**选择**：所有打卡相关分享 `imageUrl` 必须是合成的卡片图临时路径；非打卡场景（如纯角落详情分享）使用预置封面图。
**理由**：「歌词 + 我的照片」是创意核心，分享封面是最有效的传播载体。
**备选**：默认小程序 LOGO — 失去仪式感。

### D8：UGC 角落审核 — 运营人工审核 + 状态机
**选择**：用户提交进入 `pending` → 运营在企微管理后台审核 → 通过 `approved` 后批量发布到下一版 `corners.json`；驳回 `rejected` 并附理由。
**理由**：MVP 阶段量级可控（< 1000 条/月）；防止低质 / 违规内容污染主数据。
**备选**：自动审核（图像识别 + 文本分类）— 模型采购与准确率风险，留待 v2。

### D9：比赛交付形态 — 交互式 HTML Demo（单文件自包含）
**选择**：以 `index.html` 单文件（内联 CSS + JS + 种子数据）作为比赛评审交付物，Zip 打包上传社区；图片走 CDN（Unsplash 真实地点匹配图）。
**理由**：比赛要求"交互式可体验 HTML 格式文件"，双击即开、零安装、评审可点击体验是最高优先级；单文件避免路径断裂风险；浏览器 Canvas/localStorage/Web Share API 足以复刻小程序核心体验。
**实现映射**：手机外壳（375×812）+ TabBar + hash 路由页面栈；`map`→SVG 自绘地图；`canvas 2d`→浏览器 Canvas；`wx.chooseMedia`→`<input type=file>`；`wx.setStorage`→`localStorage`；`onShareAppMessage`→Web Share API + 复制兜底；暗号核销→模拟 token 二维码 + 模拟商家核销按钮。
**备选**：Taro H5 编译产物 — 体积大、需起本地服务器、路径依赖多，不适合"双击即开"评审场景，否决。

## Risks / Trade-offs

- **R1：本地存储 2MB 上限** → 评论 / 足迹采用「本地 50 条 + 云端归档」滚动策略；用户登录态下自动归档；未登录用户超额时提示登录。
- **R2：Canvas 跨端字体差异** → 仅使用系统字体（PingFang SC / 思源黑体），不嵌入网络字体；上线前 iOS/Android 各 3 款主流机型真机测试。
- **R3：LBS 偏移** → 双圈策略（D5）+ 标记"现场/附近"；不参与"严格徽章"的发放。
- **R4：Token 被刷 / 暗号滥用** → 云函数原子幂等（unique index `userOpenidHash + cornerId + dateKey`）；单用户日限 1 次；TTL 10 分钟；商家端核销日志保留 90 天。
- **R5：CDN JSON 缓存击穿** → 文件名加 `_meta.version`（D3 输出）+ 小程序冷启动版本对比；CDN 端配置较短 max-age（300s）。
- **R6：歌词版权审计** → `lyric-data-pipeline` 强制 ≤ 30 字 + `lyric_credit` 字段；CI 阻断超长片段；运营每月例行抽查。
- **R7：合作商家撤销 / 倒闭** → `corners.json` 提供 `status: active|paused|closed` 字段，小程序端对 `closed` 显示"已歇业"且不可打卡。
- **R8：演唱会信息延迟** → CDN JSON 配合企业微信群机器人推送热修复；前端缓存 1 小时；网络失败显示"已缓存：{时间}"。
- **R9：分享率不达预期** → 通过 A/B 测试不同卡片模板与文案；演唱会节点 KOL 五迷投放；未达 30% 时优先优化卡片视觉而不是上新功能。

## Migration Plan

由于这是新建产品，无既有用户，不涉及"迁移"；但有"上线发布"步骤：

1. **数据基线**：执行 `scripts/sync-lyrics-snippets.py` + `scripts/build-corners.py` 生成第一版数据，CI 通过合规校验。
2. **CDN 发布**：通过 `scripts/publish-cdn.sh` 上传 `_meta.version=1` 的 JSON。
3. **云开发部署**：部署 3 个云函数 + 创建 4 个集合 + 配置安全规则。
4. **小程序提审**：分包加载（主包 ≤ 2MB）；填写隐私协议；提交微信审核。
5. **首批合作店铺接入**：商家分包灰度，5-10 家店主完成商家账号绑定。
6. **灰度上线**：先开放至首期 5 个城市；监控指标 7 天；通过后扩展至下一批 3-5 城。
7. **回滚策略**：CDN 保留前 N 个版本（`corners.v{n-1}.json`），如新版数据问题，1 分钟内切换回上一版本；云函数支持版本回滚。
